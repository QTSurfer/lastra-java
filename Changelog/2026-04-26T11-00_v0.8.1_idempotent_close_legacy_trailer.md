# lastra-java 0.8.1 — idempotent close + legacy 4-byte trailer reader

## Summary

Two correctness fixes for the writer/reader round-trip.

### Writer: `LastraWriter.close()` is now idempotent

`close()` used to rebuild and emit the full file body on every
invocation. Callers that called `close()` more than once (for
example, an explicit `close()` inside a try-with-resources block,
followed by the implicit close on resource unwind) would silently
get the body written twice — yielding a file with two complete
Lastra blobs concatenated together.

After this change, `close()` no-ops on every call past the first.
Regression test: `LastraRoundtripTest.testCloseIsIdempotent`.

### Reader: handle the legacy 4-byte FOOTER_MAGIC trailer

`LastraWriter` was extended (after 0.8.0) to append a 4-byte
footer-size hint after `FOOTER_MAGIC`, so HTTP-Range-style clients
can locate the footer in two requests. Files written by 0.8.0 and
older consumers don't have that hint — their trailer is just the
4-byte magic.

`LastraReader`'s row-groups branch assumed the hint was always
present and read `data[length - 4]` as `footerSize`. On a legacy
file that's the magic itself (≈ 1.28e9), so `footerPos` came out
massively negative and the constructor threw
`ArrayIndexOutOfBoundsException`.

The reader now detects whether the hint is present and falls back
to a forward-scan when it isn't: it walks the data section in
`seriesColCount`-sized groups of length-prefixed columns and
locates the footer by matching the remaining byte budget against
the closed-form footer length for the current row-group count.

Regression test:
`LastraRoundtripTest.testRowGroupsWithLegacyFourByteTrailer`.

## Migration

Consumers should bump `<lastra.version>` to `0.8.1`. Files written
by the buggy `close()` path (visible as a doubled file size with
two `TSAL` magic markers, one at offset 0 and one at offset
`size/2`) can be repaired by truncating to `size/2` — both halves
are byte-identical, so the first half is a complete, well-formed
Lastra blob.
